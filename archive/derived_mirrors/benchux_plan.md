## Purpose

`ARIS BenchUX Lab` is a future product-reality macroblock intended to validate whether ARIS is not only technically strong, but also competitive, practical, reliable, visually credible, demonstrable, and usable by real clients.

BenchUX must not replace Bedrock. It must produce market, UX, interface, dashboard, client-experience, reliability, and demo-readiness evidence for later Crisol refinement and Bedrock decision.

## Short Definition

```text
BenchUX = Benchmark + Product UX + Interface + Dashboard + Client Experience + Reliability + Demo Readiness
```

Canonical expanded name:

```text
ARIS BenchUX Lab — Product Reality, Interface & Client Experience Validation
```

## Why BenchUX Exists

ARIS can be architecturally advanced and still fail as a product if:

- users do not understand what it does;
- users do not know what they are approving;
- users cannot create automations practically;
- the dashboard does not expose operational state clearly;
- evidence and rollback are technically present but visually unclear;
- the interface feels fragile, ugly, slow, or improvised;
- the system freezes, bugs, or creates anxiety during key flows;
- competitors are simpler, clearer, or more useful in real workflows;
- the demo cannot show value in a short, convincing sequence.

BenchUX exists to prevent technical excellence from becoming product theater.

## BenchUX Pillars

### 1. Market Benchmark

Compare ARIS against real alternatives and references, including at minimum:

- n8n;
- Zapier;
- Make;
- Retool;
- Temporal;
- LangGraph;
- CrewAI;
- AutoGen;
- OpenAI Agents SDK;
- Claude/Codex workflows;
- MCP ecosystem;
- traditional RPA;
- relevant devtools and operational platforms such as Linear, Vercel, GitHub, Stripe, Datadog, and Sentry where useful for UX/product patterns.

Benchmark must answer:

- where ARIS is clearly better;
- where ARIS is worse;
- where ARIS is only different;
- what competitors do more simply;
- what competitors lack in evidence, rollback, auditability, governance, local-first behavior, or safety;
- what ARIS must not copy;
- what ARIS should learn from.

### 2. Client Reality

Evaluate ARIS from the perspective of a real client, not an engineer.

Questions:

- Does the client understand the value quickly?
- Can the client explain what ARIS does after seeing it?
- Can the client imagine using it in a real operation?
- Does the product reduce anxiety or increase it?
- Does the client trust it enough to approve an action?
- Does the client understand the result and the evidence?
- Does the client see time saved, errors reduced, or control improved?

BenchUX must consider both micro and macro business contexts, such as barbershops, offices, stores, agencies, internal operations, service businesses, and larger operational teams.

### 3. Creation Experience

ARIS must be practical to create with.

BenchUX must evaluate:

- creating a task;
- creating a workflow;
- editing a workflow;
- duplicating a workflow;
- using templates;
- asking ARIS to propose an automation;
- previewing before execution;
- simulating before approval;
- authorizing safely;
- understanding frequency, scope, risk, rollback, and expected output.

Bad creation experience:

```text
The user must understand action runtime, gates, JSON, schemas, and artifacts before creating value.
```

Good creation experience:

```text
The user asks for an operational outcome; ARIS creates a clear plan with permissions, risk, preview, evidence expectation, and rollback.
```

### 4. Operational Dashboard

BenchUX must define and validate a real product dashboard.

The dashboard should expose:

- active workflows;
- pending approvals;
- recent executions;
- failed executions;
- rollback availability;
- generated evidence;
- risk level;
- time saved;
- cost/time estimates;
- system health;
- permissions granted;
- next scheduled operations;
- warnings and degraded-mode state;
- audit trail.

The dashboard must be client-readable. It must not be only a technical log viewer.

### 5. Approval, Evidence, and Rollback UX

The user must be able to understand:

- what ARIS will do;
- why it wants permission;
- what can go wrong;
- what will not happen;
- what evidence will be generated;
- how rollback works;
- whether rollback is available;
- what happened after execution;
- what failed and what to do next.

The UI must avoid exposing internal complexity unless the user asks for details.

### 6. Reliability Experience

UX includes stability. BenchUX must test whether ARIS feels reliable during real use.

Scenarios:

- loading states;
- slow operation;
- failed operation;
- retry;
- refresh during operation;
- duplicate click;
- network unavailable;
- local runtime unavailable;
- partial execution;
- rollback unavailable;
- evidence generation failure;
- stale state;
- empty state;
- session expiration;
- response truncation;
- frontend/backend mismatch;
- frozen or unresponsive interface.

A technically safe system still fails BenchUX if it looks broken to the user.

### 7. Visual Quality and Premium Feel

BenchUX must evaluate whether the interface feels like a serious product.

Criteria:

- visual hierarchy;
- spacing;
- typography;
- contrast;
- responsive layout;
- dark/light behavior if applicable;
- cards and tables;
- dashboard layout;
- state indicators;
- microcopy;
- empty states;
- error states;
- animation restraint;
- premium feel;
- visual consistency with ARIS identity.

A product that handles risk, permission, and automation must visually communicate control, calm, trust, and professionalism.

### 8. Demo and Sales Readiness

BenchUX must prove whether ARIS can be demonstrated simply.

Minimum demo sequence:

```text
1. User requests an automation.
2. ARIS creates a plan.
3. ARIS shows risk, permissions, and expected evidence.
4. User approves.
5. ARIS executes or simulates according to authorized scope.
6. ARIS shows result and evidence.
7. ARIS shows rollback or performs rollback if authorized.
8. Dashboard records the lifecycle.
```

If ARIS cannot show value in a two-minute demo, it is not product-real yet.

## Minimum PASS Criteria

BenchUX should not pass unless evidence shows:

- ARIS was compared against relevant market alternatives.
- ARIS differentiation is clear and defensible.
- Areas where ARIS loses are explicitly recorded.
- The creation flow is practical for a non-engineer client.
- The dashboard communicates operational state clearly.
- Approval UX is understandable.
- Evidence UX is visible and useful.
- Rollback UX is understandable.
- Failure states are clear and non-catastrophic.
- The product does not freeze, bug, or feel fragile in critical flows.
- Visual quality feels serious and premium enough for the target market.
- A two-minute demo can show value without architecture explanation.
- Remaining gaps are routed to Crisol or Bedrock as explicit residual risk.

## Minimum FAIL Criteria

BenchUX should fail or block if:

- ARIS cannot be differentiated from simpler tools;
- competitor products solve the user journey more clearly;
- users cannot create a workflow without technical knowledge;
- approval/evidence/rollback are confusing;
- dashboard state is missing or unreadable;
- core UI flows freeze, desynchronize, or fail silently;
- failure UX creates user anxiety or false confidence;
- visual quality damages trust;
- demo requires long explanation to make sense;
- product claims are made without Bedrock authorization.

## Interaction with Crisol

BenchUX should generate findings for Crisol.

Crisol should refine:

- simplicity;
- interface polish;
- onboarding;
- demo narrative;
- evidence visibility;
- rollback clarity;
- dashboard quality;
- product language;
- client-facing maturity.

Technical correctness alone should not be enough for Crisol PASS if BenchUX findings remain unresolved or unaccepted.

## Interaction with Bedrock

Bedrock should not authorize productization unless BenchUX evidence is satisfactory or residual BenchUX risks are explicitly accepted.

Bedrock must treat the following as product-readiness blockers unless waived by explicit evidence:

- no competitive benchmark;
- unclear differentiation;
- unusable creation flow;
- confusing approval UX;
- unclear rollback UX;
- weak evidence UX;
- fragile dashboard;
- poor reliability experience;
- UI that does not communicate trust;
- no convincing demo path.

## Research Needed Before Execution

Before materializing BenchUX as an active macroblock, run a deep research pass on:

- competitive benchmarking methodology for automation and agent platforms;
- UX patterns in automation tools, devtools, dashboards, approval flows, and incident systems;
- onboarding and demo-readiness practices in technical products;
- dashboard design for auditability, permissions, risk, evidence, and rollback;
- reliability UX and failure-state design;
- premium interface systems for trust-heavy products;
- product-readiness gates before pilot/commercialization.
