# BENCHUIX-27A Visual Sandbox Contract

## status

BENCHUIX-27A is an artifact-only substep of BENCHUIX-27. It defines the ARIS Visual Sandbox contract before any React static prototype exists.

- Layer name: ARIS Visual Sandbox.
- Nature: visual sandbox, candidate-only, synthetic-only.
- Preferred future format: React static prototype.
- This phase does not write executable React code.
- This phase does not create `package.json`.
- This phase does not execute npm or any package manager.
- This phase does not run a local preview.
- This phase does not call API, backend, fetch, XHR, WebSocket, OAuth, PSP, billing, database, auth, secrets or runtime.
- This phase does not mutate `Project_ARIS`.
- This phase does not admit CRISOL.

## purpose

The visual sandbox must turn the BenchUIX documentary layer into an observable synthetic experience without claiming product readiness. The contract exists so a later prototype can be built from stable visual rules instead of improvising runtime-like behavior.

## required screens

1. Hoje
2. Aprovar
3. Historico / Comprovantes
4. Rollback / Desfazer
5. Falha / Modo degradado

Every screen must show `SandboxBadge` and the statement that no real state was touched.

## required scenarios

1. Barbearia: after-hours booking is blocked and a VIP exception is approved by the operator.
2. Mercado: suspected refund is declined with evidence.
3. Escritorio: suspicious attachment is intercepted and routed to human review.

All scenarios are synthetic. Scenario data must never include real customer, real payment, real document, real schedule, real auth or real integration identifiers.

## central UI rule

Every sensitive action must show:

- what ARIS will do;
- what ARIS will not do;
- which risk exists;
- whether approval is needed;
- which evidence will be generated;
- whether rollback or compensation is available;
- whether real state was touched.

The visual primitive that carries this rule is `ActionContractCard`.

## ActionContractCard contract

`ActionContractCard` is mandatory on every screen that presents a sensitive action, blocked action, approval, refusal, receipt, rollback or degraded state.

It must contain:

- `WillDoList`
- `WillNotDoList`
- `RiskBadge`
- `ApprovalRequirement`
- `EvidencePreview`
- `RollbackStatus`
- `StateTouchedIndicator`
- `SandboxBadge`

The component must never hide will-not, state touched, evidence or rollback status behind secondary navigation.

## screen behavior

### Hoje

Purpose: show what deserves attention now.

Must display:

- scenario switcher;
- today summary;
- pending sensitive item;
- current visual state;
- sandbox badge;
- state touched false;
- quick access to Aprovar and Historico.

### Aprovar

Purpose: let the operator approve, refuse, keep blocked or send to review only after seeing the full action contract.

Must display:

- `ActionContractCard`;
- decision options;
- decision consequence preview;
- evidence preview;
- rollback or compensation status;
- disabled execution-looking controls in degraded mode.

### Historico / Comprovantes

Purpose: prove what happened in the synthetic story.

Must display:

- receipt list;
- receipt detail;
- policy/risk/evidence summary;
- verification code marked synthetic;
- will/will-not snapshot;
- state touched false.

### Rollback / Desfazer

Purpose: explain whether the synthetic action is reversible, compensable or only documentable.

Must display:

- current decision;
- rollback or compensation boundary;
- what the sandbox story will change;
- what no real system will change;
- evidence generated by rollback or compensation.

### Falha / Modo degradado

Purpose: fail closed when the sandbox cannot safely present an action.

Must display:

- degraded banner;
- reason in user language;
- blocked action list;
- no execution-looking CTA;
- evidence-only next step;
- state touched false.

## data contract

Every scenario item must include:

- `scenario_id`
- `domain`
- `event`
- `will_do`
- `will_not_do`
- `risk`
- `approval_required`
- `decision_options`
- `evidence_plan`
- `receipt_example`
- `rollback_or_compensation`
- `state_touched=false`
- `synthetic_only=true`

## state machine constraints

- ARIS may propose.
- ARIS must not auto-approve sensitive actions.
- Between proposal and visual execution there must be an operator or deterministic rule gate.
- Refusal and automatic block must generate a receipt.
- Degraded mode blocks any action that looks like execution.
- In the sandbox, `state_touched` is always false.

## absolute prohibitions

The ARIS Visual Sandbox must not use API, backend, database, real auth, `fetch()`, XHR, WebSocket, secrets, PSP real, OAuth real, billing real, real data, real runtime, `Project_ARIS` mutation, ready-product claims, public product deploy, Bedrock authorization or CRISOL admission.

## exit boundary

BENCHUIX-27A can only produce documentary artifacts and validator/test coverage. A future React static prototype requires a new explicit candidate phase and must keep this contract as input, not as proof of product readiness.
